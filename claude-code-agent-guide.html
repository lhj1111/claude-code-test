team-lead 중심 오케스트레이션 frontend / backend / db / tester /design /new-feature /review 자연어 요청도 가능
가장 중요한 원칙 작업을 직접 구현하는 에이전트보다, 먼저 어떤 에이전트에게 맡길지 고르는 것이 더 중요합니다.
초보자 추천 사용법 대부분은 team-lead에게 자연어로 요청하고, 필요하면 스킬을 제안받아 실행하는 방식이 가장 쉽습니다.
핵심 흐름 복잡한 신규 앱은 /design → /new-feature, 작은 변경은 전용 스킬이나 단일 에이전트로 처리합니다.
전체 개념

먼저 이해해야 할 구조

이 프로젝트의 Claude Code 사용 방식은 "아무 에이전트나 직접 부르기"보다는 역할을 나눈 팀 구조에 가깝습니다. 중심에는 team-lead가 있고, 실제 구현은 전문 에이전트들이 담당합니다.

team-lead 모든 요청의 진입점입니다. 요청을 해석하고, 어떤 에이전트에게 맡길지 결정하고, 결과를 통합 검증합니다.
frontend React 페이지, 컴포넌트, CSS Modules, TanStack Query, 라우터 등록, Home 허브 카드 추가를 담당합니다.
backend Hono 라우트, shared Zod 스키마, API 응답 형식, 입력 검증, 백엔드 버그 수정을 담당합니다.
db Prisma 모델, schema.prisma, 관계 설정, 마이그레이션 실행, DB 스키마 관련 작업을 담당합니다.
tester 코드를 고치지 않고 API 테스트, 빌드 검증, 통합 시나리오 확인만 수행하는 읽기/실행 전용 에이전트입니다.
문서 구조

이 프로젝트의 md 파일들은 어떻게 작용하나

이 프로젝트에서 md 파일은 단순 참고 문서가 아니라, Claude Code가 작업 방식을 이해하고 반복 흐름을 실행하는 기준점 역할을 합니다. 특히 CLAUDE.md, GUIDE.md, .claude/agents/*.md, .claude/commands/*.md는 각각 역할이 다릅니다.

CLAUDE.md 프로젝트의 기본 컨텍스트 문서입니다. 기술 스택, 디렉토리 구조, API/DB/Frontend 컨벤션, 새 기능 추가 흐름, 스킬 목록 같은 프로젝트의 공통 규칙을 담고 있습니다. 문서상 모든 세션에 자동 로드되는 기준 문서로 설명됩니다.
GUIDE.md 사람이 읽는 운영 가이드에 가깝습니다. 어떤 상황에서 어떤 스킬을 쓰는지, Subagent와 Agent Team을 언제 쓰는지, 큰 흐름을 어떻게 조합하는지 설명하는 문서입니다.
.claude/agents/*.md 각 에이전트의 역할 정의서입니다. 예를 들어 team-lead.md는 조율 방식과 팀 워크플로우를, frontend.md는 프론트엔드 작업 규칙을, backend.md와 db.md는 담당 경로와 구현 컨벤션을 설명합니다.
.claude/commands/*.md 슬래시 커맨드 스킬 정의서입니다. 예를 들어 /new-feature는 어떤 순서로 스키마, API, 페이지를 만들지 적혀 있고, /design은 어떤 질문을 던져서 설계 문서를 만들지 정리돼 있습니다.
.claude/designs/*.md /design 실행 결과로 쌓이는 설계 산출물입니다. 구현 전에 요구사항, 데이터 모델, API 초안, 화면 구조를 남겨 두는 문서라고 보면 됩니다.
정리하면, CLAUDE.md는 프로젝트의 기본 규칙, GUIDE.md는 운영 안내서, .claude/agents/*.md는 역할 설명서, .claude/commands/*.md는 실행 절차서라고 이해하면 가장 쉽습니다.
빠른 선택 가이드

무엇을 써야 할지 먼저 고르는 방법

처음 읽는 사람은 에이전트 목록보다 "지금 어떤 방식을 써야 하는가"를 먼저 알면 훨씬 덜 헷갈립니다. 아래 기준으로 먼저 고른 뒤 세부 내용을 보면 됩니다.

단일 에이전트 한 레이어에서 끝나는 작업에 적합합니다. 예: @frontend로 페이지 수정, @backend로 API 수정, @db로 스키마 변경.
Subagent 특정 파일만 건드리는 단순 작업, 읽기 전용 분석, 기술 조사처럼 메인 작업을 보조하는 독립 업무에 적합합니다. 결과만 메인으로 돌아옵니다.
Agent Team 새 기능 전체 구현, 여러 레이어를 동시에 진행해야 하는 작업, 병렬 조사와 상호 소통이 필요한 작업에 적합합니다. 팀원 간 메시지와 역할 분담이 핵심입니다.

단일 에이전트 추천 상황

  • Note 페이지 UI만 고치기
  • 특정 라우트의 응답 포맷 수정
  • Prisma 컬럼 하나 추가

Subagent 추천 상황

  • 버그 원인만 먼저 조사
  • 라이브러리 비교 리서치
  • 특정 컴포넌트만 제한적으로 개선

Agent Team 추천 상황

  • 새 앱 또는 새 기능 전체 개발
  • API와 UI를 동시에 진행
  • 여러 가설로 버그 병렬 조사
간단히 말하면, "한 사람이 한 구역만 고치면 단일 에이전트", "보조 조사면 Subagent", "여러 사람이 역할을 나눠 같이 끝내야 하면 Agent Team"입니다.

현재 만들어져 있는 에이전트

@team-lead

작업 분배, 라우팅, 병렬 개발 조율, 최종 통합 검증 담당.

  • 새 기능 전체 추가 시 기본 진입점
  • 자연어 요청에서 적절한 스킬을 감지
  • 필요하면 frontend/backend/db를 순차 또는 병렬로 배치
오케스트레이터 직접 코딩보다 조율 중심

@frontend

apps/web/src/ 하위 작업 전담.

  • 페이지 추가
  • 컴포넌트 추가/수정
  • TanStack Query 연결
  • App.tsx 라우팅, Home 카드 반영
React CSS Modules

@backend

apps/api/src/, packages/shared/src/ 하위 작업 전담.

  • Hono 라우트 구현
  • Zod 스키마 작성
  • 응답 포맷 유지
  • 유효성 검증과 에러 처리
Hono Zod

@db

apps/api/prisma/ 하위 작업 전담.

  • Prisma 모델 추가/수정
  • relation 설계
  • 마이그레이션 실행
  • DB 변경 영향 검토
Prisma Migration

@tester

  • API curl 테스트
  • 빌드 확인
  • 라우트 자동 탐색 후 검증

스킬 목록

현재 준비된 슬래시 커맨드

스킬은 반복 업무를 정해진 절차대로 실행하게 해주는 명령입니다.

/design

복잡한 기능이나 새 앱을 설계부터 정리합니다.

  • 요구사항 인터뷰
  • 데이터 모델 초안
  • API 명세 초안
  • 화면 구조 정리

/new-feature

새 기능을 풀스택으로 스캐폴딩합니다.

  • shared schema
  • Prisma 모델
  • API 라우트
  • 페이지 + Home 카드

/add-component

프론트엔드 화면이나 컴포넌트를 추가합니다.

  • 페이지 생성
  • CSS Modules 생성
  • 라우터 등록

/add-endpoint

새 API 엔드포인트를 추가합니다.

  • Zod 스키마
  • Prisma 모델
  • Hono CRUD 라우트

/db-migrate

Prisma 마이그레이션 생성과 실행을 도와줍니다.

  • Docker 상태 확인
  • migrate dev 실행
  • 영향 파일 점검

/review

보안, 성능, UX 기준으로 병렬 리뷰를 수행합니다.

  • Security 체크
  • Performance 체크
  • UX 체크

/fix-bug

버그를 레이어별로 분석하고 관련 에이전트에게 수리 작업을 맡깁니다.

  • 문제 레이어 파악
  • 최소 수정 원칙
  • 검증까지 포함

/new-skill

새 슬래시 커맨드를 추가하고 관련 문서까지 업데이트합니다.

  • 동작 인터뷰
  • 명령 파일 생성
  • CLAUDE.md / GUIDE.md 반영
언제 무엇을 쓰는가

상황별 추천 사용법

새 앱 전체를 처음 만들고 싶다 /design으로 설계부터 시작하고, 설계가 확정되면 /new-feature로 구현을 여는 흐름이 가장 좋습니다.
기존 앱에 기능 하나 추가 대부분 /new-feature로 바로 시작하면 됩니다. 범위가 UI만이면 /add-component, API만이면 /add-endpoint가 더 빠릅니다.
버그가 났는데 원인이 불명확 /fix-bug 또는 @team-lead에게 자연어로 설명하는 방식이 안전합니다.
DB 컬럼/테이블 변경 @db 또는 /db-migrate를 사용합니다. 데이터 손실 가능성이 있으면 먼저 설계를 확인해야 합니다.
최종 점검이 필요 /review로 리뷰하고, 필요하면 @tester에게 API와 빌드 검증까지 맡길 수 있습니다.
실전 흐름

실제로는 이렇게 요청하면 됩니다

1
가장 쉬운 시작법: team-lead에게 자연어로 말하기

할 일 앱에 마감일 기능 추가해줘 또는 유튜브 요약 앱에 검색 필터가 안 먹어, 원인 찾아서 고쳐줘처럼 말하면 됩니다.

2
team-lead가 적절한 스킬을 제안하거나 에이전트를 선택

큰 기능이면 /design 또는 /new-feature, UI면 frontend, DB면 db 식으로 라우팅합니다.

3
전문화된 에이전트가 작업 수행

새 기능은 보통 db → backend → frontend 순서로 흐르고, 필요하면 병렬 작업도 가능합니다.

4
필요하면 tester나 review로 마무리

구현 후 API 응답, 빌드, UX 품질까지 다시 점검하면 안정적입니다.

처음에는 개별 에이전트를 직접 다루기보다, @team-lead 또는 적절한 스킬부터 쓰는 편이 훨씬 덜 헷갈립니다.
고급 사용법

현재 프로젝트에서 Agent Team을 어떻게 쓰면 좋은가

이 프로젝트는 큰 작업을 여러 에이전트가 나눠 처리하도록 설계되어 있습니다. 특히 team-lead 문서에는 새 기능 전체 구현은 팀 모드로 다루는 것을 기본 패턴으로 보고 있고, 실제 순서도 db → backend → frontend를 기준으로 잡고 있습니다.

가장 적합한 상황 새 앱 또는 새 기능 전체 구현, API와 UI를 동시에 진행해야 하는 작업, 여러 레이어에 걸친 버그 조사처럼 한 사람이 순서대로 하기엔 긴 작업입니다.
굳이 팀이 필요 없는 상황 단순 페이지 수정, API 한 개 추가, Prisma 컬럼 하나 추가 같은 단일 레이어 작업은 단일 에이전트가 더 빠르고 단순합니다.
핵심 운영 원칙 team-lead는 조율만 하고, 각 팀원은 자기 담당 경로의 파일만 수정해야 합니다. 같은 파일을 두 팀원이 동시에 만지는 것은 피해야 합니다.
1
tmux 기반 팀 모드를 시작합니다

가이드 문서 기준 시작 명령은 claude --teammate-mode split입니다. 문서상 전제는 WSL + tmux 환경이며, VS Code 통합 터미널이나 일반 Windows Terminal은 권장 환경이 아닙니다.

2
team-lead가 팀을 구성합니다

보통 db, backend, frontend를 팀원으로 붙이고, 검증이 중요하면 마지막에 tester를 추가합니다.

3
작업은 순서와 의존성을 명확히 둡니다

새 기능 기준으로는 먼저 DB/스키마가 끝나야 backend가 안정적으로 진행되고, backend가 API를 열어야 frontend가 타입과 계약에 맞춰 안전하게 붙일 수 있습니다.

4
팀원끼리 완료 신호를 직접 주고받습니다

예를 들어 db가 완료되면 backend에게, backend가 완료되면 frontend에게 시작 가능 신호를 보내는 흐름입니다.

5
마지막은 team-lead가 통합 검증합니다

shared schema re-export, API 응답 형식, App 라우터 등록, Home 카드 추가, 타입 오류 여부까지 확인하고 팀을 정리합니다.

추천 팀 구성

  • team-lead: 요청 분석, 태스크 분해, 최종 검증
  • db: Zod 스키마, Prisma 모델, migration
  • backend: Hono 라우트, 검증, 응답 포맷
  • frontend: 페이지, Query/Mutation, 라우터, Hub 카드
  • tester: 필요 시 CRUD/API/build 검증

파일 소유권 예시

  • db: apps/api/prisma/
  • backend: apps/api/src/, packages/shared/src/
  • frontend: apps/web/src/
  • 같은 파일에 대해 동시 수정은 피하기

팀원 간 소통이 가능한가?

가능합니다. 현재 프로젝트의 team-lead 문서에는 팀 모드에서 에이전트들이 SendMessage로 서로 직접 소통하는 흐름이 명시돼 있습니다.

  • db가 완료 후 backend에게 시작 신호 전달
  • backend가 완료 후 frontend에게 API 준비 알림
  • 각 팀원이 team-lead에게 진행 상황과 완료 보고

어떤 종류의 소통이 가능한가?

  • 작업 시작 요청: "이제 네 작업 시작해도 돼"
  • 블로커 공유: "스키마가 아직 안 나와서 backend 대기 중"
  • 세부 확인: "필드 이름을 pinned로 갈지 isPinned로 갈지 확인 필요"
  • 리뷰 요청: "API 응답 형식 한번 확인해줘"
  • 완료 보고: "라우트 구현 완료, 프론트 진행 가능"
A
단순 병렬 실행만 하는 구조가 아닙니다

현재 설계는 각 팀원이 독립적으로 따로 도는 것이 아니라, 의존성이 있는 작업은 메시지로 연결하면서 이어서 진행하는 협업 구조입니다.

B
리뷰 요청을 끼워 넣는 것도 가능합니다

예를 들어 backend가 API 응답 형태를 만든 뒤 frontend에게 타입 관점 확인을 요청하거나, team-lead에게 통합 관점 리뷰를 요청하는 식으로 진행할 수 있습니다.

C
다만 최종 조율자는 team-lead입니다

팀원끼리 직접 소통은 가능하지만, 전체 우선순위와 충돌 해결, 최종 완료 판단은 team-lead가 맡는 구조로 보는 게 가장 정확합니다. 

Create an agent team to build a "bookmark" feature.

- Teammate 1 (db): packages/shared/src/schemas/bookmark.ts 생성,
  Prisma 모델 추가, migration 준비

- Teammate 2 (backend): apps/api/src/routes/bookmark/ 구현,
  routes/index.ts 등록, 응답 형식 컨벤션 유지

- Teammate 3 (frontend): apps/web/src/pages/Bookmark/ 구현,
  App.tsx 등록, Home 카드 추가

- Optional Teammate 4 (tester): CRUD API와 web build 검증
          
예시 소통 흐름

db → backend:
"shared schema와 migration 완료. 이제 라우트 구현 시작해도 돼."

backend → frontend:
"POST/GET/PATCH/DELETE 응답 형식 맞춰서 구현 완료. 페이지 연결 진행해줘."

frontend → team-lead:
"화면 연결 완료. Home 카드와 App 라우터 등록까지 끝남. 통합 확인 부탁."

team-lead → tester:
"bookmark 기능 전체 흐름과 build 상태 검증해줘."
네, 현재 구조에서는 팀원끼리 서로 소통하고, 시작 요청이나 완료 알림, 검토 요청을 주고받으면서 진행할 수 있습니다. 다만 그 흐름을 안정적으로 묶고 최종 판단을 내리는 중심은 여전히 team-lead입니다.

추천 패턴

가장 실무적으로 좋은 사용 순서

신규 앱 개발

  • /design으로 요구사항 정리
  • /new-feature로 구현 시작
  • 완료 후 /review로 점검

작은 기능 추가

  • UI만이면 /add-component
  • API만이면 /add-endpoint
  • DB 변경 동반 시 /db-migrate 연결

버그 수정

  • /fix-bug 또는 @team-lead
  • 원인 미확정 상태에서 frontend/backend/db를 바로 찍지 않아도 됨

품질 보증

  • /review로 병렬 관점 점검
  • @tester로 실행 검증까지 보강
활용 사례

현재 프로젝트 기준 실제 활용 시나리오

사례 1. 새 앱 전체 개발

예: 북마크 앱, 지출 관리 앱처럼 화면, API, DB가 모두 필요한 경우.

  • 먼저 /design으로 요구사항 정리
  • 그다음 /new-feature 또는 Agent Team 구성
  • db → backend → frontend 흐름으로 구현 후 /review와 @tester로 점검

사례 2. 기존 앱에 API + UI 동시 추가

예: Note 앱에 pinned 필드와 필터 UI를 동시에 붙여야 하는 경우.

  • @team-lead에게 자연어로 요청
  • db가 필드 추가, backend가 API 변경
  • frontend가 UI와 Query 연결

사례 3. 버그 원인 불명확

예: "저장 후 목록이 갱신되지 않음"처럼 프론트인지 백인지 모호한 버그.

  • /fix-bug 또는 @team-lead 사용
  • 필요하면 여러 에이전트가 가설별로 병렬 조사
  • 원인이 좁혀지면 해당 레이어 에이전트가 수정

사례 4. 읽기 전용 조사

예: TanStack Query 설정을 SWR로 바꿀지 비교만 먼저 보고 싶은 경우.

  • Subagent로 조사만 수행
  • 코드 변경 없이 장단점 요약 수집
  • 결정 후 실제 구현은 단일 에이전트나 팀으로 진행

사례 5. 품질 점검 단계

예: 기능은 구현됐지만 실제로 안전한지, 빠른지, UX가 괜찮은지 확인하고 싶은 경우.

  • /review로 보안/성능/UX 병렬 리뷰
  • @tester로 API와 build 검증
  • 수정 필요 항목만 다시 관련 에이전트에 전달

사례 6. 아주 작은 UI 수정

예: Todo 페이지 버튼 정렬, CSS Modules 스타일 보정, 문구 수정.

  • 이 경우는 Agent Team까지 갈 필요 없음
  • @frontend 또는 /add-component로 충분
  • 작업 비용 대비 가장 효율적
