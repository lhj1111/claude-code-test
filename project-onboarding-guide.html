이 프로젝트는 작은 실험용 페이지를 하나씩 붙여 가는 단순한 앱 모음이 아닙니다. 프론트엔드, API, 데이터베이스, 공유 스키마를 같은 규칙으로 묶어 두고, 새 기능이나 새 앱을 추가할 때도 같은 흐름으로 확장하도록 설계된 풀스택 모노레포입니다.

React 18 + Vite Hono API Prisma + PostgreSQL shared Zod schema pnpm workspace
이 프로젝트를 한 줄로 말하면 새 앱을 빠르게 붙일 수 있도록 규칙화된 React + API + DB 모노레포입니다.
처음 이해해야 할 핵심 프론트와 백엔드는 따로 움직이지만, 둘의 계약은 shared Zod 스키마가 맡습니다.
신규 앱 개발의 출발점 복잡하면 설계부터, 명확하면 스키마부터 시작합니다. UI부터 바로 만들지 않습니다.
프로젝트 성격

이 프로젝트는 무엇을 하려고 만들어졌나

홈 화면에서 여러 실험 앱으로 이동할 수 있고, 각 앱은 독립적인 화면을 가지면서도 같은 개발 규칙을 따릅니다. 현재 문서와 코드 기준으로 Todo, Note, Data Visualizer, YouTube Summary 같은 앱이 이미 포함되어 있습니다.

1. 실험용 앱을 계속 추가하기 쉬움

앱 하나를 만들 때 필요한 기본 골격이 정해져 있어서, 새 주제가 생겨도 같은 방식으로 붙일 수 있습니다.

2. 프론트와 백의 약속이 명확함

응답 형식, 상태 코드, 스키마 위치가 정해져 있어 어느 레이어를 봐야 할지 빠르게 판단할 수 있습니다.

3. AI 협업에 맞춘 운영 구조

문서, 커맨드, 에이전트 역할이 구분되어 있어 반복 작업을 스캐폴딩하기 좋은 구조입니다.

구조 이해

폴더를 이렇게 보면 금방 이해됩니다

핵심은 세 레이어입니다. 화면, API, 그리고 둘이 공유하는 계약입니다.

apps/web 사용자가 보는 화면 React, Vite, React Router, TanStack Query
→
apps/api 데이터를 주고받는 서버 Hono, Prisma, 라우트별 엔드포인트
→
PostgreSQL 지속 저장이 필요한 데이터 Prisma schema와 migration으로 관리

프론트엔드 기준 폴더

페이지와 공통 UI가 들어갑니다. 새 앱 화면도 여기서 시작합니다.

apps/web/src/pages/ apps/web/src/components/ apps/web/src/App.tsx

백엔드 기준 폴더

기능별 API 라우트가 모입니다. 라우트 등록 파일도 별도로 있습니다.

apps/api/src/routes/ apps/api/src/index.ts apps/api/src/db/

공유 계약 폴더

프론트와 백엔드가 같이 쓰는 Zod 스키마와 타입을 둡니다.

packages/shared/src/schemas/ packages/shared/src/index.ts
핵심 규칙

이 프로젝트에서 꼭 지켜야 하는 규칙

API 응답 형식 성공은 { data: T }, 실패는 { error: { message, code } } 형식을 사용합니다.
엔드포인트 이름 /api/{resource} 형태의 복수 명사 리소스를 기본으로 사용합니다.
Prisma 모델 규칙 모델명은 PascalCase 단수, 필드는 camelCase, 기본 필드 id, createdAt, updatedAt를 포함합니다.
페이지 위치 새 페이지는 보통 apps/web/src/pages/{Name}/index.tsx 패턴을 따릅니다.
상태 관리 서버 데이터는 TanStack Query로 가져오고, 생성/수정/삭제 후에는 invalidate 패턴을 사용합니다.
공유 스키마 새 리소스의 Zod 스키마는 packages/shared/src/schemas/에 만들고 index.ts에서 re-export 해야 합니다.
신규 앱 개발 흐름

새 앱이나 새 기능을 만들 때의 실제 순서

이 프로젝트에서는 화면부터 바로 만드는 방식보다, 데이터 계약을 먼저 잡고 그 위에 API와 화면을 올리는 흐름을 권장합니다.

1
먼저 범위를 정합니다

요구사항이 크거나 모호하면 /design으로 설계 문서를 만들고, 단순하면 바로 구현 단계로 갑니다.

2
shared Zod 스키마를 만듭니다

새 앱의 핵심 엔티티와 Create/Update 입력 스키마를 packages/shared/src/schemas/{name}.ts에 정의합니다.

3
Prisma 모델과 마이그레이션을 준비합니다

apps/api/prisma/schema.prisma에 모델을 추가하고 마이그레이션을 실행해 DB 구조를 맞춥니다.

4
Hono 라우트를 만듭니다

apps/api/src/routes/{name}/index.ts에 CRUD나 필요한 API를 구현하고, 라우트 통합 파일에 등록합니다.

5
React 페이지를 만듭니다

페이지를 만들고 Query/Mutation으로 API를 연결한 뒤, 로딩·에러·빈 상태까지 처리합니다.

6
라우터와 홈 허브를 연결합니다

apps/web/src/App.tsx에 경로를 등록하고, Home 페이지 카드에 새 앱을 추가해 진입점을 만듭니다.

7
검증합니다

타입 체크, 빌드, 그리고 실제 사용 흐름 기준의 QA까지 확인한 뒤 마무리합니다.

이 흐름의 핵심은 "스키마가 먼저, UI는 나중"입니다. 프론트와 백엔드는 같은 shared schema를 중심으로 움직입니다.
자주 쓰는 커맨드

작업을 빠르게 시작할 때 사용하는 명령

/design

복잡한 기능이나 앱을 설계부터 정리하고 싶을 때 사용합니다. 요구사항 인터뷰 후 설계 문서를 생성합니다.

/new-feature

새 기능을 풀스택 기준으로 추가할 때 사용합니다. 스키마, API, 페이지, 허브 연결 흐름이 기준입니다.

/add-endpoint

기존 앱에 API만 추가하거나 백엔드 작업이 중심일 때 적합합니다.

/add-component

새 화면, 컴포넌트, 프론트 UI 작업이 중심일 때 사용합니다.

/db-migrate

Prisma 스키마 변경 후 마이그레이션을 생성하고 적용할 때 사용합니다.

/review

보안, 성능, UX 관점에서 병렬로 점검할 때 사용합니다.

실행 방법

로컬에서 프로젝트를 띄우는 순서

처음 세팅할 때와 평소 개발할 때의 명령은 크게 다르지 않습니다.

pnpm install
docker compose up -d
pnpm db:migrate
pnpm dev
개발 서버를 띄우면 보통 웹은 5173, API는 3000 포트를 사용하고, PostgreSQL은 Docker Compose로 구동합니다.
처음 보는 사람에게 주는 팁

어디부터 읽으면 가장 빨리 적응하나

1. Home 페이지 보기

이 프로젝트에 현재 어떤 앱이 있는지, 허브 카드 구조가 어떤지 먼저 보면 감이 빨리 옵니다.

2. routes/index 보기

현재 API가 어떤 단위로 묶여 있는지 보면 백엔드 구조를 바로 이해할 수 있습니다.

3. shared schema 보기

프론트와 백이 어떤 계약을 공유하는지 파악하면 새 기능 추가 흐름이 훨씬 선명해집니다.

이 문서는 프로젝트 입문자가 구조, 규칙, 신규 앱 개발 흐름을 빠르게 이해할 수 있도록 요약한 온보딩 자료입니다.
